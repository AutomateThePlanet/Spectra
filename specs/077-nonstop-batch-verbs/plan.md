## Research Findings (Code-Confirmed)

### Finding 1 — `IngestVerdictCommand.cs` is advisory-gate only (no writes)
`src/Spectra.CLI/Commands/Generate/IngestVerdictCommand.cs` — only has `--from <file>` (optional, reads stdin if omitted). Uses `VerdictIngestor.Classify(content)` which is a static, model-free, no-I/O classifier. Returns `{verdict, score, drop}`. Persists nothing.

**Batch mode shape**: Enumerate `.spectra/verdicts/critic-verdict-*.json`, filter by suite via `_index.json` lookup (same as `ingest-grounding`), classify each file, aggregate counts `{grounded, partial, hallucinated, errors}`. Return summary JSON.

### Finding 2 — `IngestUpdateCommand.cs` is per-test-only (no staging path)
`src/Spectra.CLI/Commands/Generate/IngestUpdateCommand.cs` — takes `suite` (positional) + `--test-id` (required) + `--from` (optional). Calls `UpdatedTestIngestor.IngestAsync(content, testsPath, suite, originalTest, existingTests)`. There is **no `.spectra/updates/` staging path in the current codebase** — it reads the edited content directly from `--from` or stdin.

**Batch mode shape**: Requires defining a new staging convention: `.spectra/updates/{suite}/updated-{id}.json`. The skill writes each repaired test to this path (using Write tool), then calls `ingest-update {suite} --all` once. The batch command enumerates `updated-*.json` files, extracts `id` from filename, loads original test from suite, calls `UpdatedTestIngestor.IngestAsync()` per file.

### Finding 3 — `IngestGroundingCommand.cs` is the exact template
`src/Spectra.CLI/Commands/Generate/IngestGroundingCommand.cs:72-163` (batch path): enumerate `.spectra/verdicts/critic-verdict-*.json`, filter by `_index.json` lookup, process each, emit `{written, skipped_already_written, skipped_partial_pre_repair, errors}`. This is the pattern to mirror.

### Finding 4 — No `.spectra/updates/` path exists yet
The `.spectra/updates/` path only exists in the spec documents. This spec introduces it as a new convention. It must be defined and documented clearly (naming: `updated-{id}.json`, gitignored alongside `.spectra/verdicts/`).

## Key Design Decisions

### `ingest-verdict --all` output shape
Mirrors `ingest-grounding --all` but with verdict counts instead of write counts:
```json
{
  "command": "ingest-verdict",
  "mode": "batch",
  "suite": "unit-converter",
  "grounded": 12,
  "partial": 3,
  "hallucinated": 2,
  "errors": 0
}
```

### `ingest-update --all` staging convention
New staging path: `.spectra/updates/{suite}/updated-{id}.json`
- Created by the agent's repair loop (Write tool)
- Consumed by `ingest-update {suite} --all`
- Gitignored (alongside `.spectra/verdicts/`)
- Naming mirrors verdict files: `updated-TC-100.json` ↔ `critic-verdict-TC-100.json`

```json
{
  "command": "ingest-update",
  "mode": "batch",
  "suite": "checkout",
  "written": 5,
  "skipped_no_staging_file": 0,
  "errors": 0
}
```

### Skill change (8b repair loop)
**Current (per-entry):**
```
For each manifest entry:
  → compile-repair-prompt → agent patches → ingest-update {suite} --test-id {id} --from repaired.json
```
**New (write-then-batch):**
```
For each manifest entry:
  → compile-repair-prompt → agent patches → Write to .spectra/updates/{suite}/updated-{id}.json
After all repairs:
  → ingest-update {suite} --all
```

### `--from` and `--all` mutual exclusion
Both verbs must error if both `--from` and `--all` are passed. `IngestUpdateCommand` also requires `--test-id` when using `--from`, but `--test-id` is irrelevant (and must be absent or ignored) when using `--all`.
