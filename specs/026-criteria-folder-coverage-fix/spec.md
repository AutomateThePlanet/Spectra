# Feature Specification: Criteria Folder Rename, Index Exclusion & Coverage Fix

**Feature Branch**: `026-criteria-folder-coverage-fix`  
**Created**: 2026-04-08  
**Status**: Draft  
**Input**: User description: "Criteria Folder Rename, Index Exclusion & Coverage Fix"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Criteria folder uses consistent naming (Priority: P1)

A user who has run `spectra ai analyze --extract-criteria` has criteria files stored under `docs/requirements/`. This contradicts the terminology used everywhere else in the tool (CLI flags, dashboard labels, class names), creating confusion. After this fix, all criteria files live under `docs/criteria/`, and existing projects are migrated automatically.

**Why this priority**: The folder naming inconsistency is the most visible bug — every user interacting with the file system encounters it. It also blocks the other two fixes since path resolution affects both extraction exclusion and coverage calculation.

**Independent Test**: Run `spectra ai analyze --extract-criteria` on a fresh project and verify output goes to `docs/criteria/`. Run on a project with existing `docs/requirements/` and verify automatic migration.

**Acceptance Scenarios**:

1. **Given** a new project initialized with `spectra init`, **When** the user inspects the created directory structure, **Then** `docs/criteria/` exists and `docs/requirements/` does not.
2. **Given** an existing project with `docs/requirements/` containing criteria files, **When** the user runs any `spectra ai analyze` command, **Then** `docs/requirements/` is renamed to `docs/criteria/` and all files are preserved.
3. **Given** an existing project with `docs/requirements/` and a `spectra.config.json` containing hardcoded `"docs/requirements"` paths, **When** migration runs, **Then** the config file is updated to reference `"docs/criteria"`.
4. **Given** an existing project where both `docs/requirements/` and `docs/criteria/` already exist, **When** migration runs, **Then** no action is taken and a warning is logged.
5. **Given** an existing project with neither `docs/requirements/` nor `docs/criteria/`, **When** migration runs, **Then** no error occurs and no folders are created.

---

### User Story 2 - Index metadata files excluded from criteria extraction (Priority: P1)

A user runs `spectra ai analyze --extract-criteria` and the tool processes `docs/_index.md` — a metadata file generated by `spectra docs index` — producing `_index.criteria.yaml` with ~113 duplicate criteria. After this fix, `_index.md` and similar metadata files are automatically skipped during extraction.

**Why this priority**: This bug causes duplicate criteria IDs, inflated coverage counts, wasted AI calls, and nonsensical `component: _index` entries. It directly corrupts data quality.

**Independent Test**: Run `spectra ai analyze --extract-criteria` on a project with `docs/_index.md` and verify no `_index.criteria.yaml` is created.

**Acceptance Scenarios**:

1. **Given** a docs directory containing `_index.md` alongside real documentation files, **When** the user runs `spectra ai analyze --extract-criteria`, **Then** `_index.criteria.yaml` is not created and `_index.md` is not processed.
2. **Given** a docs directory containing `_index.yaml` or `_index.json` metadata files, **When** extraction runs, **Then** these files are skipped.
3. **Given** a docs directory containing existing `.criteria.yaml` files, **When** extraction runs, **Then** criteria files are not re-extracted from themselves (no circular extraction).
4. **Given** an existing `_index.criteria.yaml` from a previous extraction, **When** the migration/cleanup logic runs, **Then** the file is deleted and the master index is rebuilt without its entries.

---

### User Story 3 - Dashboard shows correct acceptance criteria coverage (Priority: P1)

A user has extracted criteria and generated tests with `criteria:` frontmatter linking them. The dashboard shows "0 of 0 acceptance_criteria covered" despite valid data existing. After this fix, the dashboard accurately reflects criteria coverage.

**Why this priority**: The dashboard is the primary visibility tool for test coverage. Showing zero coverage when criteria exist undermines user trust and makes the criteria feature appear broken.

**Independent Test**: Extract criteria, generate tests with criteria linkage, run `spectra dashboard`, and verify the acceptance criteria section shows correct totals.

**Acceptance Scenarios**:

1. **Given** 5 criteria across two `.criteria.yaml` files and 2 tests referencing criteria IDs in their frontmatter, **When** the user generates a dashboard, **Then** the acceptance criteria section shows "2 of 5 covered" with a 40% progress bar.
2. **Given** criteria files exist but no tests reference any criteria IDs, **When** the user generates a dashboard, **Then** the section shows "0 of 5 covered" (not "0 of 0").
3. **Given** no `docs/criteria/` directory exists, **When** the user generates a dashboard, **Then** the section shows "0 of 0" with a helpful empty-state message guiding the user to run `spectra ai analyze --extract-criteria`.
4. **Given** two tests both reference the same criterion ID, **When** coverage is calculated, **Then** that criterion counts as covered once (not twice).
5. **Given** a criterion exists only in a per-document `.criteria.yaml` file (not in the master `_criteria_index.yaml`), **When** coverage is calculated, **Then** that criterion is still counted in the total.

---

### Edge Cases

- What happens when `docs/requirements/` contains files not related to criteria (e.g., user-created notes)? Migration moves the entire folder, preserving all files.
- What happens when `_index.md` is nested inside a subdirectory (e.g., `docs/subdir/_index.md`)? It is still excluded from extraction — the exclusion checks the filename, not the full path.
- What happens when a test references a criterion ID that does not exist in any `.criteria.yaml` file? The criterion is not counted (orphaned reference). No error is raised.
- What happens when `spectra.config.json` has custom (non-default) criteria paths? The migration only rewrites paths containing `"docs/requirements"`. Custom paths are preserved.
- What happens when `.criteria.yaml` files exist inside `imported/` subdirectory? They are included in the coverage count — imported criteria are real criteria.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST default criteria storage to `docs/criteria/` for all new projects.
- **FR-002**: System MUST automatically rename `docs/requirements/` to `docs/criteria/` when the old folder exists and the new one does not, upon running any `spectra ai analyze` command.
- **FR-003**: System MUST update `spectra.config.json` paths from `"docs/requirements"` to `"docs/criteria"` during migration.
- **FR-004**: System MUST NOT migrate when both `docs/requirements/` and `docs/criteria/` exist; a warning MUST be logged instead.
- **FR-005**: System MUST skip `_index.md`, `_index.yaml`, `_index.json`, `.criteria.yaml` files, and `_criteria_index.yaml` during criteria extraction.
- **FR-006**: System MUST delete any existing `_index.criteria.yaml` file during the migration cleanup step.
- **FR-007**: System MUST read all per-document `.criteria.yaml` files from the configured criteria directory when calculating acceptance criteria coverage.
- **FR-008**: System MUST match test frontmatter `criteria:` field against criterion IDs to determine coverage status.
- **FR-009**: System MUST count each criterion as covered if at least one test references it, regardless of how many tests reference it.
- **FR-010**: System MUST populate the dashboard's acceptance criteria section with correct total, covered, and uncovered counts from the criteria coverage analyzer.
- **FR-011**: System MUST display an actionable empty-state message in the dashboard when no criteria exist, directing users to run `spectra ai analyze --extract-criteria`.
- **FR-012**: System MUST preserve all files during folder migration, including non-criteria files that may exist in `docs/requirements/`.
- **FR-013**: All SKILL files and agent prompts MUST reference `docs/criteria/` instead of `docs/requirements/` in path examples.

### Key Entities

- **Criterion**: A testable acceptance criterion extracted from documentation or imported from external tools. Stored in per-document `.criteria.yaml` files with an ID (e.g., `AC-CHECKOUT-001`), title, and optional linked test IDs.
- **Criteria Directory**: The folder (`docs/criteria/` by default) containing all per-document `.criteria.yaml` files, the master `_criteria_index.yaml`, and an `imported/` subdirectory.
- **Master Index**: `_criteria_index.yaml` — an auto-generated aggregation of all criteria across per-document files. Used for quick lookup but not the sole source of truth for coverage.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: All new projects created with `spectra init` use `docs/criteria/` with zero references to `docs/requirements/` in generated files.
- **SC-002**: Existing projects with `docs/requirements/` are automatically migrated on first command run, with all files preserved and config paths updated.
- **SC-003**: Criteria extraction produces zero criteria files for metadata documents (`_index.md`, `_index.yaml`, `_index.json`).
- **SC-004**: Dashboard acceptance criteria section shows accurate coverage counts matching the actual number of criteria in `.criteria.yaml` files and the number of tests referencing them.
- **SC-005**: Coverage report totals match the sum of criteria across all per-document `.criteria.yaml` files (excluding `_criteria_index.yaml` and `_index.criteria.yaml`).
- **SC-006**: All existing tests continue to pass after the rename, with path-dependent tests updated to use the new directory name.

## Assumptions

- The `docs/requirements/` folder has no user-created content unrelated to criteria that would be confused by renaming. If such content exists, it is acceptable to move it along with criteria files since the folder is documented as criteria storage.
- The master `_criteria_index.yaml` is regenerated from per-document files, so it does not need special migration handling beyond moving it with the folder.
- The `_index.md` exclusion applies to all directories, not just the top-level `docs/` folder, since subdirectories may also have index files.
- The primary source of truth for criteria coverage is the test frontmatter `criteria:` field (Path A), not the `linked_test_ids` field in `.criteria.yaml` files (Path B).
