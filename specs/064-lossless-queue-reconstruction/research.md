# Phase 0 Research: Lossless Execution-Queue Reconstruction

All decisions below are resolved. The two material questions were settled in `/speckit.clarify`
(recorded in `spec.md` → Clarifications); the rest are grounded in a direct read of the execution
engine, queue, repositories, and the existing reconstruction tests.

## Decision 1 — Source of truth for reconstruction: persisted snapshot (not index re-read)

**Decision**: Persist a durable orchestration snapshot (`title`, `priority`, `depends_on`,
`order_index`) per queued test at run-build time, in a new `queue_snapshot` table. Reconstruction
reads only the DB (snapshot + `test_results`), never the on-disk `_index.json`.

**Rationale**:
- FR-001 demands behaviour "identical to the **original** in-memory queue." The original queue was
  built from the index *as it was at `StartRunAsync` time* (`TestQueue.Build` consuming
  `TestIndexEntry.Title/Priority/DependsOn`, `TestQueue.cs:29-50`). The index is mutable mid-run
  (regenerated by `generate`/`update` flows), so re-reading it at reconstruction time can yield
  different values → divergence from the original queue. Only a build-time snapshot guarantees
  fidelity.
- Makes reconstruction **DB-complete** (Q1 of the investigation: today the queue is "derivable from
  disk but not from the DB alone" — this closes that gap), so the fail-loud surface shrinks to genuine
  corruption rather than "index moved."
- Aligns with Constitution II (deterministic execution: same inputs → same queue).

**Alternatives considered**:
- **Re-read the on-disk index at reconstruction** (the spec's original informed-guess default):
  rejected — zero schema change but reintroduces silent divergence on any mid-run index edit, directly
  contradicting FR-001.
- **Re-read index + drift fingerprint that fails loud on mismatch**: rejected — still couples
  reconstruction to a mutable external file and turns a benign, legitimate index edit into a hard
  failure for an otherwise-healthy run; more moving parts than a write-once snapshot.

## Decision 2 — Where the snapshot lives: a new `queue_snapshot` table (not columns on `test_results`)

**Decision**: A dedicated table `queue_snapshot(run_id, test_id, title, priority, depends_on,
order_index)` with `PRIMARY KEY (run_id, test_id)` and `FOREIGN KEY (run_id) REFERENCES runs(run_id)
ON DELETE CASCADE`.

**Rationale**:
- Orchestration data is invariant **per `(run_id, test_id)`**, but `test_results` has **one row per
  attempt** (`PRIMARY KEY (run_id, test_id, attempt)`, `ExecutionDb.cs:87`). Putting orchestration on
  `test_results` would duplicate it across retest attempts and invite inconsistency. A per-test table
  is the natural normal form and matches the "orchestration snapshot" entity in the spec.
- Mirrors the existing one-table-one-repository pattern (`runs`→`RunRepository`,
  `test_results`→`ResultRepository`) — no new abstraction (Constitution V).
- `ON DELETE CASCADE` keeps snapshot lifecycle tied to the run, identical to `test_results`.

**Alternatives considered**:
- **Add `title`/`priority`/`depends_on`/`order_index` columns to `test_results`**: rejected —
  per-attempt duplication, and `order_index` is meaningless per attempt.
- **Serialize the whole ordered queue as one JSON blob on `runs`**: rejected — opaque to queries,
  harder to validate per-test, and diverges from the row-oriented storage style already in use.

## Decision 3 — Fail-loud mechanism: throw a distinct typed exception

**Decision**: Introduce `QueueReconstructionException` (in `Spectra.MCP/Execution/`). Reconstruction
throws it when a run has recorded results but its snapshot is missing, incomplete (a result's
`test_id` has no snapshot row, or a snapshot test has no result), or internally inconsistent (a
snapshot `depends_on` points at a `test_id` absent from the snapshot). The benign "run has no results
/ run not found" case continues to return `null` (non-throwing absence). Tool handlers translate the
exception to a distinct error code (e.g. `RECONSTRUCTION_FAILED`), never `RUN_NOT_FOUND`.

**Rationale**:
- Consistent with the engine's existing throw-on-invalid-state behaviour (`AdvanceTestAsync` throws
  `InvalidOperationException` on an unknown handle / invalid transition, `ExecutionEngine.cs:208-217`).
- Keeps "absent" and "corrupt" as distinguishable signals — required by FR-003 and testable for
  SC-003 via `Assert.Throws`.

**Alternatives considered**:
- **Discriminated result type** (`Reconstructed`/`RunNotFound`/`SnapshotCorrupt`): rejected — diverges
  from the established throw pattern and forces a touch of every caller's control flow.
- **Reuse the null path with an error code**: rejected — weakest guarantee; a caller could treat it as
  benign "not found."

## Decision 4 — Unify the queue-access paths through reconstruct-aware `GetQueueAsync`

**Decision**: Route `GetStatusAsync`, `AdvanceTestAsync`, `BulkRecordResultsAsync`, `RetestAsync`, and
`FinalizeRunAsync`'s pending-guard through `GetQueueAsync` (which returns the cached `_queues` entry
when present, else reconstructs). Remove the direct `_queues.TryGetValue(...)` reads that silently
no-op when the queue is absent.

**Rationale**:
- This is the single change that realises FR-004 (long-lived == short-lived) and the investigation's
  6→0 B-column collapse. `GetQueueAsync` already prefers the in-memory queue, so the long-lived path
  is unchanged (FR-005); the short-lived path now gets a faithful queue.
- Fixes the `retest` cross-process latent bug for free: `GetStatusAsync` returns `null` today when the
  queue is absent (`ExecutionEngine.cs:114-117`), causing `retest_test_case` to fail `RUN_NOT_FOUND`
  after a restart (investigation Open Unknown #3).

**Alternatives considered**:
- **Leave callers reading `_queues` and only fix `ReconstructQueue`**: rejected — the lossy data would
  be fixed but the callers that bypass reconstruction (`GetStatusAsync`/`RetestAsync`/`Finalize`)
  would still degrade or hard-fail across process boundaries.

## Decision 5 — Remove the lossy `TestQueue.AddFromResult`; migrate its unit tests

**Decision**: Delete `AddFromResult` (the primitive that hard-codes `Title=TestId`,
`Priority=Medium`, `DependsOn=null`) and add a lossless reconstruction builder that takes ordered
snapshot entries + the latest result per test. Migrate the 8 tests in
`TestQueueReconstructionTests.cs` to the new builder (they assert queue mechanics — `GetNext`,
`MarkCompleted`, `CurrentIndex`, `CompletedCount` — which are preserved, and will additionally now
assert orchestration preservation).

**Rationale**:
- `AddFromResult` is the bug. A `TestResult` structurally cannot carry orchestration data, so the
  method cannot be made lossless; keeping it is a footgun that invites re-introducing silent loss
  (Constitution V: no shims when code can be changed directly).
- These tests are in `Spectra.MCP.Tests`, **not** in the protected regression net (`Spectra.Core` /
  `TestPersistenceService`). The investigation explicitly flags this file as "exercises the lossy
  reconstruction path; read this first before any reconstruction change" — i.e. it is expected to
  change. This is the one deviation from the spec's "Rewrite: none expected" prediction; it is a
  migration to a stronger assertion, not a weakening.

**Note**: `Spectra.Core` tests and `TestPersistenceServiceTests` are **not touched** (regression gate).
Existing long-lived-behaviour engine/integration tests (`BlockingCascadeTests`, `RetestFlowTests`,
`ExecutionFlowTests`, etc.) stay green because the warm `_queues` path is unchanged.

## Decision 6 — `StartRunAsync` snapshot persistence is part of run creation (FR-007)

**Decision**: In `StartRunAsync`, after `TestQueue.Build`, persist the snapshot rows (derived from the
built queue's ordered `Tests`: `order_index` = list position, plus `Title`/`Priority`/`DependsOn`)
before/with the result records. If snapshot persistence fails, the exception propagates and run
creation fails — no run is left unreconstructable.

**Rationale**: Directly from FR-002/FR-007. The snapshot is the sole reconstruction source, so a run
without one is unreconstructable; creating it must be a precondition of a usable run.

**Open follow-on (out of scope, noted)**: Wrapping run + snapshot + results in a single SQLite
transaction for full atomicity is desirable but belongs with the concurrency/atomicity hardening of
the consolidation spec; today's separate writes match the existing `StartRunAsync` style and the
fail-loud requirement is still met (a failed snapshot write throws).

## Legacy data note

Runs created **before** this feature have no `queue_snapshot` rows. If such a run is still in-flight
when the code is upgraded and its in-memory queue is gone, reconstruction will fail loud (correct per
FR-003). The `.execution/spectra.db` is local, ephemeral execution state; finish or cancel in-flight
runs before upgrading. Documented in quickstart.
